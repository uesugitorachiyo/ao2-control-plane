## What's new

### Live audit-log SSE tail (`GET /api/v1/audit-log/stream`)

v0.1.10 and v0.1.11 gave operators a Prometheus surface for
audit-log volume + persistence retention. The polling JSON
surface (`/api/v1/audit-log`) and the operator dashboard
(`/api/v1/audit-log/dashboard[.json]`) already covered
post-hoc inspection. What was still missing: a live tail
that can drive real-time dashboards, on-call pagers, and
external SIEMs without polling every few seconds and
hammering the ring-buffer mutex.

v0.1.12 ships `GET /api/v1/audit-log/stream` — a
Server-Sent Events surface (`Content-Type: text/event-stream`)
that streams audit-log entries as they happen.

#### Wire format

Each entry is one SSE event block:

```
event: audit-log
id: 1748284825493712
data: {"timestamp_unix_micros":1748284825493712,"method":"GET",...}

```

- `event: audit-log` — discriminator so clients can switch on
  event type via EventSource's `addEventListener("audit-log", ...)`.
- `id: <timestamp_unix_micros>` — the entry's wall-clock
  timestamp, which doubles as a monotonic-on-this-process
  cursor. Browsers preserve this as `lastEventId` and
  replay it via the `Last-Event-ID` request header on
  automatic reconnect.
- `data:` — the same `ao2.cp-audit-log.v1` entry shape that
  `/api/v1/audit-log` returns in its `entries[]` array. Bearer
  tokens are never copied into the payload (same redaction as
  the polling JSON surface).

#### Filters

Accepts the same query params as `/api/v1/audit-log`:
`method`, `status`, `status_class`, `path_prefix`,
`authenticated`. The filter applies to both the replay tail
and the live tail, using the same `matches_filters` helper
that the polling endpoint uses (refactored to `pub(crate)`
so the two surfaces never diverge).

#### Last-Event-ID resume

Two ways to specify the resume cursor:

1. `?last_event_id=<micros>` query param (wins if both are set)
2. `Last-Event-ID: <micros>` request header (the EventSource
   default on automatic reconnect)

The handler subscribes to the broadcast channel **before**
snapshotting the ring buffer, so any entry that arrives
between snapshot and subscription is captured exactly once
(either via replay or via the live tail) — no gaps, no
duplicates. Replay emits all entries with
`timestamp_unix_micros > last_event_id` that match the
filter, then the stream switches to live tail.

#### Backpressure: `event: lagged`

The live tail is fed by a bounded `tokio::sync::broadcast`
channel (capacity 256, lossy). If a subscriber falls behind,
the broadcast layer drops the oldest in-flight entries and
notifies the receiver with `Lagged(n)`. The handler surfaces
this as:

```
event: lagged
data: {"missed":<n>}

```

so the client can fall back to
`/api/v1/audit-log?since_unix_micros=<last_id>` to backfill
the gap, then resume the stream. The append path itself never
blocks — a slow subscriber cannot back-pressure the audit-log
hot path.

#### Keepalive

The handler emits a SSE comment line every 15 s
(`KeepAlive::new().interval(Duration::from_secs(15))`). This
keeps reverse proxies + EventSource clients from declaring
the connection dead during low-traffic windows (default idle
timers are 60-120 s in nginx, Cloudflare, and most ingress
controllers).

### Implementation details

The append path gains one new line:

```rust
let _ = self.stream_tx.send(entry);
```

where `stream_tx: broadcast::Sender<AuditEntry>` is a
`broadcast::channel(256).0`. `send()` returns an error only
when there are zero subscribers, which is a no-op for our
purposes (`let _`). No mutex contention added to the append
hot path — the broadcast channel uses its own internal lock
distinct from the ring-buffer mutex.

The handler bridges `broadcast::Receiver` to a `Stream` via
`tokio_stream::wrappers::BroadcastStream`, then chains the
replay tail (`futures::stream::iter`) in front of the live
tail. The merged stream is wrapped in `axum::response::Sse`
with a 15-second keepalive.

### Suggested integration

```js
// Browser EventSource — replay everything since last_id
// and keep the live tail open. Token must be carried via
// cookie or proxy because EventSource cannot set headers.
const es = new EventSource(
  "/api/v1/audit-log/stream?last_event_id=0",
  { withCredentials: true }
);

es.addEventListener("audit-log", (ev) => {
  const entry = JSON.parse(ev.data);
  appendToDashboard(entry);
});

es.addEventListener("lagged", (ev) => {
  // Fall back to /api/v1/audit-log to backfill the gap.
  backfill(JSON.parse(ev.data).missed);
});
```

```sh
# Headless tail via curl. Bearer token via Authorization
# header. last_event_id=0 replays everything in the buffer
# before going live.
curl -N \
  -H "Authorization: Bearer $AO2_CP_API_TOKEN" \
  "$AO2_CP_BASE_URL/api/v1/audit-log/stream?last_event_id=0&method=POST"
```

## Upgrade guide

- Binaries are drop-in replacements; on-disk data layout is
  unchanged from v0.1.11 (no migration required).
- New route `GET /api/v1/audit-log/stream` is added to the
  catalog at `/api/v1/routes`. No existing routes change.
- The new SSE handler shares the
  `AO2_CP_API_TOKEN` bearer-token middleware with the rest
  of `/api/v1/*` — no new authentication knob.

## Security posture

- OAuth-only authentication; no API-key provider auth.
- Bearer tokens never appear in the SSE payload, route logs,
  or the `id`/`event` lines. Verified by
  `stream_emits_events_for_live_traffic` test.
- SSE 401s without `Authorization: Bearer <token>`. Verified
  by `stream_requires_bearer_token` test.
- The broadcast channel is bounded (256 entries). A slow
  subscriber cannot back-pressure append, cannot exhaust
  memory, and is notified of dropped entries via
  `event: lagged` so the operator side can backfill via the
  polling JSON surface.
