## What's new

### Cross-OS NDJSON shipper integration helper

v0.1.5–v0.1.8 built the audit-log story end-to-end: opt-in
NDJSON persistence (v0.1.5), persistence health on `/api/v1/status`
(v0.1.6), size-based rotation (v0.1.7), and the operator HTML +
JSON dashboard (v0.1.8). v0.1.9 closes the operator integration
loop by giving operators a turnkey way to pipe the audit-log
NDJSON into the log shipper they already run:

```
scripts/ship-audit-log.sh           # Mac/Linux (bash)
scripts/ship-audit-log.ps1          # Windows (PowerShell)
```

Both helpers copy the live audit-log NDJSON to stdout, ready
to be piped into Vector, Fluent Bit, NXLog, or a
`logrotate`-managed archive. They discover the persistence
path by reading `/api/v1/status` with the configured bearer
(set `AO2_CP_API_TOKEN` or pass `--token`/`-Token`); operators
who already know the path can pass `--path`/`-Path` and skip
the round-trip. With `--follow`/`-Follow` the helpers tail
the live file with `tail -F` (bash) or `Get-Content -Wait`
(PowerShell) so the stream keeps producing across rotation
events without operator intervention.

Example: pipe authenticated audit-log NDJSON into Vector on Linux:

```
AO2_CP_API_TOKEN=$(cat /etc/ao2-cp/token) \
  scripts/ship-audit-log.sh --include-rotated --follow \
  | vector --quiet -c /etc/vector/audit-log.toml
```

Example: pipe the live file into a Fluent Bit `stdin` input on
Windows (PowerShell):

```
$env:AO2_CP_API_TOKEN = Get-Content C:\ao2-cp\token
.\scripts\ship-audit-log.ps1 -IncludeRotated -Follow `
  | C:\fluent-bit\bin\fluent-bit.exe -i stdin -o stdout
```

### Bearer-token handling

The bearer is only forwarded as an `Authorization` header on
the `/api/v1/status` round-trip. It is never written to stdout,
stderr, or any URL. The audit-log NDJSON itself already redacts
bearer values at write time, so downstream shippers receive
token-free entries. The
`ship_audit_log_streams_live_file_to_stdout_and_redacts_token`
integration test asserts the token literal never appears on
stdout or stderr after a mix of authenticated + unauthenticated
requests.

## Upgrade guide

- Binaries are drop-in replacements; on-disk data layout is
  unchanged from v0.1.8 (no migration required).
- The new scripts are additive — no API or JSON shape changes
  between v0.1.8 and v0.1.9.
- Both helpers require `AO2_CP_API_TOKEN` (or `--token`/
  `-Token`) when the persistence path is auto-discovered;
  the `--path`/`-Path` override lets operators skip the
  round-trip entirely when the audit-log file path is
  already known to the shipping environment.

## Security posture

- OAuth-only authentication; no API-key provider auth.
- The bearer token value is never copied into the helper's
  stdout, stderr, or any URL.
  `ship_audit_log_streams_live_file_to_stdout_and_redacts_token`
  asserts this on every test run.
- `ship_audit_log_fails_when_bearer_missing` asserts the
  helper rejects with a non-zero exit code when no bearer is
  configured and the persistence path was not pre-supplied.
- `ship_audit_log_includes_rotated_sidecar_when_flag_set`
  asserts the rotated `<path>.1` sidecar is streamed in
  newest-after-oldest order when `--include-rotated` /
  `-IncludeRotated` is set.
- `ship_audit_log_path_override_skips_status_roundtrip`
  asserts the helper does not call `/api/v1/status` when the
  path is supplied explicitly — proven by pointing the helper
  at an unreachable base-url and confirming it still succeeds.
