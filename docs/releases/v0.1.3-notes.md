## What's new

### Antigravity acceptance provider (third provider pilot)
Adds end-to-end support for a third provider — Antigravity — across the
acceptance pipeline. The schema layer recognizes
`ao2.antigravity-provider-pilot-acceptance.v1`, storage gains
`BundleKind::AcceptanceAntigravity` (path `acceptance/antigravity/`), and
all server handlers (`/api/v1/acceptance`, phase-1 promotion dashboard,
release publication, provider readiness, provider registry) treat
Codex / Claude / Antigravity uniformly. The promotion dashboard and
phase-1 three-OS smoke gates now correlate Antigravity acceptance
evidence alongside the other two providers.

### Observer HEAD + ETag + 304 caching fan-out
The provider-registry caching pattern shipped in v0.1.2 now covers
**every** content-addressed observer `GET` route. Each endpoint:

- Emits a strong RFC-7232 ETag (`"<sha>"`)
- Sends `Cache-Control: public, max-age=60, must-revalidate`
- Honors `If-None-Match` with a `304 No Content` short-circuit that
  **avoids the disk read entirely**
- Has a parallel `HEAD` route emitting only the headers

Endpoints covered (10 total):

- `GET/HEAD /api/v1/acceptance/:sha`
- `GET/HEAD /api/v1/control-plane/bundle/:sha`
- `GET/HEAD /api/v1/evidence-pack/:sha`
- `GET/HEAD /api/v1/memory/export/:sha`
- `GET/HEAD /api/v1/phase1/promotion/checklist/:sha`
- `GET/HEAD /api/v1/phase1/promotion/decision/:sha`
- `GET/HEAD /api/v1/phase1/promotion/three-os-smoke/:sha`
- `GET/HEAD /api/v1/provider/readiness/:sha`
- `GET/HEAD /api/v1/release/evaluator-decision/:sha`
- `GET/HEAD /api/v1/release/publication/:sha`

Shared implementation lives in
`crates/ao2-cp-server/src/handlers/caching.rs`; regression coverage in
`tests/observer_caching.rs` verifies the four-property caching contract
per endpoint (10 tests × 4 assertions).

## Upgrade guide

- Binaries are drop-in replacements; on-disk data layout is
  unchanged from v0.1.2 (no migration required).
- New `HEAD` routes and ETag/Cache-Control headers are additive — no
  client breakage. Existing clients that ignore conditional-request
  headers continue to receive the full body.
- The Antigravity provider is opt-in; existing Codex/Claude-only
  deployments are unaffected.

## Security posture (unchanged)

- OAuth-only authentication; no API-key provider auth.
- Bearer-token middleware unchanged; new HEAD routes and the new
  Antigravity acceptance path all sit under `/api/v1/*` and inherit
  it.
- The 304 short-circuit runs **before** the disk read, so a stale
  `If-None-Match` cannot be used as a side channel to learn about
  bundle presence beyond what an authenticated client already sees.
