## What's new

### Operator binary: `ao2-cp-gc`
A standalone garbage-collector binary now ships next to `ao2-cp-server`.
It is the apply-mode counterpart to the existing `/api/v1/storage/prune`
dry-run endpoint, for operators who prefer cron over HTTP. See
[`docs/runbooks/storage-retention.md`](../runbooks/storage-retention.md)
for trust boundary, cron/launchd/Task Scheduler scheduling, exit-code
contract, and `--keep-latest` sizing guidance. Bounded-growth smoke is
enforced in CI via `tests/gc_smoke.rs` (6 tests; 6 green on each of
Mac/Linux/Windows).

### TLS reverse-proxy templates (Caddy + nginx)
Drop-in `Caddyfile.example` and `nginx/ao2-cp-server.conf.example`
templates with TLS termination, per-IP rate limits (120 req/min for
`/api/v1/*`, 600 req/min for `/healthz`), HSTS, and a header-strip rule
that blocks `OPENAI_API_KEY` / `ANTHROPIC_API_KEY` headers from
reaching the upstream. Pairs with the server's existing preflight
refusal to start with those keys in its environment. See
[`deploy/README.md`](../../deploy/README.md#tls-reverse-proxy-templates).

### Prometheus `/api/v1/metrics` endpoint
Bearer-token-gated Prometheus text-exposition v0.0.4 endpoint emitting:

- `ao2_cp_requests_total{method,status_class}` — request counters (20
  fixed-cardinality buckets, lock-free `AtomicU64`)
- `ao2_cp_request_duration_seconds_{sum,count}` — request latency
  cumulative sum + count (pair to derive average)
- `ao2_cp_in_flight_requests` — current in-flight gauge (RAII guard)
- `ao2_cp_storage_total_index_entries` — observer-evidence count

Implemented without pulling in `metrics` / `metrics-exporter-prometheus`
to preserve the minimal dependency graph. See
[`deploy/README.md#prometheus-scraping`](../../deploy/README.md#prometheus-scraping)
for scrape-config example.

### Structured `/api/v1/status` endpoint
Token-gated `ao2.cp-status.v1` JSON document covering build info
(version, target triple, profile from `build.rs`), storage stats
(index entries, recursive data-dir byte count), retention pressure
percentage against the 100k-entry soft cap, request totals + error
rate + in-flight, and process uptime. Distinct from `/healthz` /
`/readyz` (binary probes, no auth) — this is the
dashboard-and-operator read of the same underlying state.

### Provider registry HEAD + ETag + 304 caching
`GET` / `HEAD` on `/api/v1/provider/registry/{latest,:sha}` now emit a
strong ETag (the quoted SHA-256 of the JCS-canonical payload) plus
`Cache-Control: public, max-age=60, must-revalidate`. Matching
`If-None-Match` short-circuits to `304 No Content`. Prereq for AO2
evidence pollers running over high-latency or cellular links.

### Structured access-log middleware
Every HTTP request emits one JSON line to stderr (target
`ao2_cp_server::access`) with `method`, `path`, `status`,
`duration_micros`, `auth_attempted`, `authenticated`. The bearer token
value is **never** read or copied — only its presence is recorded.
Verified by an integration test that asserts the literal token bytes
never appear in captured log output.

### Developer tooling: `scripts/smoke-three-os.sh`
Fast cross-OS smoke for the dev iteration loop — packages
`git archive HEAD`, ships to Ubuntu (`AO2_CP_UBUNTU_SSH_TARGET`) and
Windows (`AO2_CP_WINDOWS_SSH_TARGET`, ProxyJump-capable), runs
`cargo test -p ao2-cp-server` on all three, emits one
`ao2.cp-smoke-three-os.v1` JSON summary plus per-OS logs. The existing
`smoke-three-os-release.sh` remains the canonical pre-release gate;
this is the lighter-weight counterpart for everyday development.

## Upgrade guide

- Binaries are drop-in replacements; the on-disk data layout is
  unchanged from v0.1.1.
- New `ao2-cp-gc` binary is bundled in each archive at
  `bin/ao2-cp-gc[.exe]`; copy alongside `ao2-cp-server` to enable cron
  pruning.
- New `/metrics`, `/status`, HEAD-on-registry, and access-log
  endpoints/behaviors are additive — no client breakage.

## Security posture (unchanged)

- OAuth-only authentication; no API-key provider auth.
- Bearer-token middleware unchanged; new endpoints all sit under
  `/api/v1/*` and inherit it.
- Reverse-proxy templates strip `OPENAI_API_KEY` / `ANTHROPIC_API_KEY`
  headers from inbound requests as defense-in-depth against accidental
  fan-out.

## Cross-OS verification

All commits in this release are verified on macOS aarch64, Linux
x86_64, and Windows x86_64 via `scripts/smoke-three-os.sh`. Test
counts on the new functionality (commit-by-commit, all green on all
