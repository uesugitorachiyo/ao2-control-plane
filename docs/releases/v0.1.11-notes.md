## What's new

### Audit-log persistence + retention gauges

v0.1.10 shipped four monotonic counters that fire *after*
something has gone wrong (rotation, drop, persistence error)
or that simply measure volume (appended). v0.1.11 adds the
two missing predictive signals so operators can alert
*before* something goes wrong:

```
# TYPE ao2_cp_audit_log_file_bytes gauge
ao2_cp_audit_log_file_bytes <N>

# TYPE ao2_cp_audit_log_oldest_resident_age_seconds gauge
ao2_cp_audit_log_oldest_resident_age_seconds <N>
```

Gauge semantics:

- **`ao2_cp_audit_log_file_bytes`** — current byte size of the
  live NDJSON persistence file. Reads the same `bytes_written`
  field on `PersistenceState` that drives the rotation decision,
  so the gauge value is always exact (not an inode-stat snapshot
  that may race against an in-flight write). Reports `0` when
  persistence is disabled (`AO2_CP_AUDIT_LOG_PERSIST_PATH` unset)
  rather than absent so the metric label is stable across
  configuration changes.
- **`ao2_cp_audit_log_oldest_resident_age_seconds`** — wall-clock
  age in seconds of the oldest entry currently resident in the
  ring buffer. Sampled at scrape time from the front of the
  `VecDeque`. Reports `0` when the buffer is empty. Future-dated
  entries (clock skew, e.g. an entry minted with a timestamp
  ahead of `SystemTime::now()`) clamp to `0` via `saturating_sub`
  so the gauge never overflows.

### Suggested Prometheus alerts

Two new alerts cover the two predictive failure modes:

```promql
# Predictive: live audit-log file is within 20% of its rotation
# threshold but rotation has not yet fired. Catches a stuck
# rotation (e.g. ENOSPC on the rotated sidecar path) before
# the file consumes the disk volume.
- alert: AuditLogFileApproachingRotation
  expr: ao2_cp_audit_log_file_bytes / <AO2_CP_AUDIT_LOG_MAX_BYTES> > 0.8
  for: 2m

# Predictive: ring buffer retention horizon dropped below the
# operator's target. Either traffic surged faster than capacity
# can hold, or persistence has been disabled and the buffer
# alone can no longer back the audit-log dashboard's lookback.
- alert: AuditLogRetentionHorizonShrinking
  expr: ao2_cp_audit_log_oldest_resident_age_seconds < <retention_target_seconds>
  for: 5m
```

Combined with the v0.1.10 counters
(`_persistence_errors_total`, `_dropped_total`,
`_rotated_total`, `_appended_total`), operators now have:

- **Reactive** alerts that fire when something already broke
  (counter rate(...) > 0)
- **Predictive** alerts that fire when something is about to
  break (gauge crosses a static threshold)

Both surfaces share the same scrape-time-sampling pattern —
no new background tasks, no new state to maintain, no new
mutex contention.

### Implementation details

`AuditLog` gains two new accessors:

- `persistence_file_bytes()`: reads the existing
  `PersistenceState.bytes_written` counter under the
  persistence mutex. Returns `0` when persistence is disabled.
- `oldest_resident_unix_micros()`: reads
  `entries.front().map(|e| e.timestamp_unix_micros)` under
  the ring-buffer mutex. Returns `None` when empty.

The `metrics_handler` samples both at scrape time and
converts the oldest-resident timestamp to fractional seconds
using saturating subtraction against `now_unix_micros()`. The
new fields ride on the same `AuditLogSamples` struct
introduced in v0.1.10.

No new state, no new mutex, no new background work, no new
configuration knobs. The two new metrics are pure derivations
of existing fields.

## Upgrade guide

- Binaries are drop-in replacements; on-disk data layout is
  unchanged from v0.1.10 (no migration required).
- The Prometheus exposition adds two new gauge families. The
  audit-log counter families from v0.1.10 are unchanged.
- The two new gauges report `0` when their underlying
  conditions are unmet (persistence disabled / buffer empty)
  rather than being absent, so Prometheus recording rules and
  alerts can reference them unconditionally.

## Security posture

- OAuth-only authentication; no API-key provider auth.
- The two new gauges expose file size and entry age — no
  request contents, no bearer tokens, no individual entries.
- `/api/v1/metrics` remains bearer-token gated by the same
  `AO2_CP_API_TOKEN` middleware as the rest of `/api/v1/*`.
