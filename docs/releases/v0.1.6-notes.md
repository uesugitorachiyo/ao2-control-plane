## What's new

### Audit-log telemetry on `/api/v1/status`

v0.1.4 shipped a bounded ring buffer surfaced by
`/api/v1/audit-log`, and v0.1.5 added opt-in NDJSON persistence
behind `AO2_CP_AUDIT_LOG_FILE`. v0.1.6 wires that subsystem into
the operator status surface so a dashboard can poll one endpoint
and get the full audit-log health picture.

The `/api/v1/status` JSON now carries an `audit_log` block:

```json
"audit_log": {
  "capacity": 1024,
  "buffered": 17,
  "total_appended_since_boot": 482,
  "persistence": {
    "enabled": true,
    "path": "/var/lib/ao2-cp/audit.ndjson",
    "last_error": null
  }
}
```

The same block (under `buffer`) is mirrored into the existing
`/api/v1/audit-log` response so polling either surface yields the
same telemetry.

Field semantics:

- `capacity` — configured ring-buffer slots
  (`AO2_CP_AUDIT_LOG_CAPACITY`, default `1024`).
- `buffered` — currently resident entries; never exceeds
  `capacity`.
- **`total_appended_since_boot`** — new monotonic counter
  (`AtomicU64`, `Relaxed` ordering, incremented after each
  successful append). Survives ring eviction, so dashboards can
  estimate request rate without keeping every entry resident.
  Resets to zero only when the server restarts; the persistence
  file (if configured) remains the durable record across restart.
- **`persistence.enabled`** — `true` when
  `AO2_CP_AUDIT_LOG_FILE` is configured.
- **`persistence.path`** — the configured file path, or `null`
  when persistence is disabled.
- **`persistence.last_error`** — most recent persistence
  write/serialise error message, or `null` when there has been
  no failure since boot. This is a **non-clearing peek** —
  distinct from the existing `take_persistence_error()`
  (clearing) used by the tracing log. Polled dashboards can
  observe error state without swallowing it before ops sees it.

### Why this matters

Persistence in v0.1.5 was deliberately self-healing: a transient
`ENOSPC` recorded `last_error` but did not disable the writer.
Without a polled surface, that recovery property masked silent
disk-full conditions — the writer kept going but ops never knew
anything had gone wrong. The v0.1.6 telemetry block closes that
loop by exposing `last_error` on `/status`, where it can be
panel-bound on a Grafana board without requiring stderr ingest.

The boot-cumulative counter is intentionally cheap (one
`AtomicU64::fetch_add` per request, `Relaxed` ordering) so that
high-throughput operators with `AO2_CP_AUDIT_LOG_CAPACITY=0`
still get an accurate request-rate signal even though no entries
are buffered in memory.

## Upgrade guide

- Binaries are drop-in replacements; on-disk data layout is
  unchanged from v0.1.5 (no migration required).
- JSON shape changes are **additive only**: existing consumers
  of `/api/v1/status` and `/api/v1/audit-log` keep parsing as
  before; the new `audit_log` block on `/status` and the new
  fields under `buffer` on `/audit-log` are pure additions.
- No new env vars or flags.

## Security posture

- OAuth-only authentication; no API-key provider auth.
- The bearer token is never copied into status or audit-log
  JSON. The new
  `status_endpoint_audit_log_block_reflects_observed_traffic`
  integration test re-asserts this by scanning the entire
  `/status` payload for the literal token string after driving
  authenticated traffic.
- `persistence.last_error` exposes an error *message*, never an
  error stack trace, file descriptor, or token bytes.
