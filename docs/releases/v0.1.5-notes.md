## What's new

### `AO2_CP_AUDIT_LOG_FILE` — opt-in NDJSON persistence

The `/api/v1/audit-log` ring buffer added in v0.1.4 gives operators a
queryable, in-process view of recent requests, but the buffer is
bounded and lost on process restart. v0.1.5 adds an opt-in file-based
mirror: set `AO2_CP_AUDIT_LOG_FILE=/path/to/audit.ndjson` and every
entry is also appended to that file as newline-delimited JSON.

- **Append-only, restart-safe.** The file is opened in
  `OpenOptions::new().create(true).append(true)` mode, so existing
  history is preserved across restart. `tail -f audit.ndjson` works
  out of the box.
- **Independent of capacity.** Persistence is decoupled from the
  ring-buffer capacity. Setting `AO2_CP_AUDIT_LOG_CAPACITY=0` keeps
  zero in memory but still mirrors every entry to disk if a file path
  is configured — useful for high-throughput operators who only want
  the durable record.
- **Flushed every entry.** A `BufWriter::flush()` runs after every
  append; a non-clean process exit loses at most the in-flight line,
  and operator `tail -f` sees entries within milliseconds.
- **Self-healing on transient errors.** Write or serialise failures
  are recorded as `last_error` (clearable via
  `take_persistence_error()`) without disabling the writer, so a
  transient `ENOSPC` does not silently disable persistence forever.
- **Token redaction is preserved end-to-end.** The serialised line
  uses the same `AuditEntry` shape as the ring buffer, which was
  already proven in v0.1.4 to never contain the `Authorization`
  header bytes. Verified by the new
  `audit_log_persistence_writes_one_ndjson_line_per_request`
  integration test (asserts the literal `TEST_API_TOKEN` substring
  never appears in any persisted line).

## Upgrade guide

- Binaries are drop-in replacements; on-disk data layout is
  unchanged from v0.1.4 (no migration required).
- `AO2_CP_AUDIT_LOG_FILE` is unset by default, so existing
  deployments are unaffected. Operators who want persistent audit
  history can opt in at any restart with a single env var.
- The ring-buffer endpoint and JSON shape are unchanged.

## Security posture

- OAuth-only authentication; no API-key provider auth.
- The persistence file inherits filesystem ACL from its parent
  directory — operators are responsible for setting `0700` on the
  containing directory if the audit log must remain confidential.
  The file itself is written with `umask`-respecting permissions.
- The persisted JSON shape is identical to the in-memory entry
  shape, so the v0.1.4 redaction guarantee (the `Authorization`
  header value is never copied into entries) extends to disk
  unchanged. Verified end-to-end by
  `audit_log_persistence_writes_one_ndjson_line_per_request`.
- The writer never reads back; it is append-only. There is no path
  by which a malformed entry can corrupt earlier lines.

## Cross-OS verification

All commits in this release are verified on macOS aarch64, Linux
x86_64, and Windows x86_64 via `scripts/smoke-three-os.sh` plus the
full release-archive smoke (`scripts/smoke-three-os-release.sh`),
which produced byte-identical per-OS surface hashes
(`release_cockpit`, `release_handoff`, `release_readiness`,
`release_publication_dashboard`, `release_assembly`). New regression
coverage:

- `crates/ao2-cp-server/src/audit_log.rs` — 7 unit tests
  (3 existing + 4 new): `persistence_writes_ndjson_one_line_per_entry`,
  `persistence_appends_across_restart_without_truncation`,
  `persistence_open_fails_when_parent_dir_missing`,
  `append_with_zero_capacity_still_persists_to_file`.
- `crates/ao2-cp-server/tests/audit_log_endpoint.rs` — 9 integration
  tests (7 existing + 2 new):
  `audit_log_persistence_writes_one_ndjson_line_per_request`,
  `audit_log_persistence_survives_two_server_lifetimes`.
