## What's new

### Size-based NDJSON rotation: `AO2_CP_AUDIT_LOG_MAX_BYTES`

v0.1.5 shipped opt-in NDJSON persistence and v0.1.6 surfaced its
health on `/api/v1/status`. The remaining gap was unbounded
file growth — a long-running operator deployment would
eventually fill its disk and trip the `persistence.last_error`
panel built in v0.1.6. v0.1.7 closes that loop with size-based
rotation:

```
AO2_CP_AUDIT_LOG_FILE=/var/lib/ao2-cp/audit.ndjson
AO2_CP_AUDIT_LOG_MAX_BYTES=10485760    # 10 MiB
```

Once the live file grows past the threshold, it is renamed to
`<path>.1` (replacing any prior sidecar) and a fresh file is
opened at the original path. There is **one** historical
generation — no `.2`, no compression, no time-based rotation.
The built-in rotation guarantees disk-usage stays bounded when
no log shipper is configured; operators who want richer retention
pipe the live NDJSON file into `logrotate`, `vector`, or
`fluent-bit` and leave `AO2_CP_AUDIT_LOG_MAX_BYTES` unset.

Cross-platform correctness details:

- The persistence writer is now `Option<BufWriter<File>>` so the
  rotation path can `take()` and drop the file handle before
  renaming. Windows rejects `rename` of an open file with
  `ERROR_SHARING_VIOLATION`; the drop closes the handle.
- The rotation routine explicitly removes any existing `<path>.1`
  before renaming. On Unix `rename` would overwrite silently;
  on Windows it would fail without the explicit `remove_file`.
- `bytes_written` is an `u64` accumulator seeded from
  `metadata().len()` at open. Restarting a process onto an
  existing file preserves the rotation cadence — the counter is
  not reset to zero every boot, so a process that frequently
  restarts cannot postpone rotation indefinitely.

### Telemetry surface

A `rotation` sub-block was added under
`audit_log.persistence` on both `/api/v1/status` and
`/api/v1/audit-log`:

```json
"persistence": {
  "enabled": true,
  "path": "/var/lib/ao2-cp/audit.ndjson",
  "last_error": null,
  "rotation": {
    "max_bytes": 10485760,
    "count": 42,
    "last_rotated_unix_micros": 1717104923000000
  }
}
```

- `rotation.max_bytes` — configured threshold, or `null` when
  rotation is disabled (`AO2_CP_AUDIT_LOG_MAX_BYTES` unset).
- `rotation.count` — total rotations since boot. Spikes above
  the operator's expected cadence indicate runaway traffic or
  a too-small threshold; zero across a busy interval indicates
  the threshold has never been reached.
- `rotation.last_rotated_unix_micros` — wall-clock micros of
  the most recent rotation, or `null` when no rotation has
  occurred this process lifetime.

## Upgrade guide

- Binaries are drop-in replacements; on-disk data layout is
  unchanged from v0.1.6 (no migration required).
- JSON shape changes are **additive only**: the new `rotation`
  sub-block under `persistence` does not break existing
  consumers of either `/api/v1/status` or `/api/v1/audit-log`.
- `AO2_CP_AUDIT_LOG_MAX_BYTES` is unset by default, so existing
  deployments are unaffected. Operators who want bounded
  growth can opt in at any restart with a single env var.

## Security posture

- OAuth-only authentication; no API-key provider auth.
- The bearer token is never copied into the live file *or* the
  rotated sidecar. The new
  `audit_log_persistence_rotates_when_threshold_exceeded`
  integration test drives 20 oversize authenticated requests
  through the live server, then reads back every line of the
  rotated sidecar and asserts the literal token string never
  appears.
- Rotation respects the same filesystem permissions as the
  live file: the rename preserves ownership and mode, and the
  fresh file is opened with `umask`-respecting permissions
  matching the original.
