## What's new

### Audit-log Prometheus counters

v0.1.5â€“v0.1.9 built the audit-log story end-to-end: opt-in
NDJSON persistence (v0.1.5), persistence health on
`/api/v1/status` (v0.1.6), size-based rotation (v0.1.7),
operator HTML + JSON dashboard (v0.1.8), and cross-OS NDJSON
shipper helpers (v0.1.9). v0.1.10 closes the telemetry loop
by exposing audit-log health as Prometheus counters so
operators can alert on silent breakage without polling
`/api/v1/status`.

Four new monotonic counters render on `/api/v1/metrics`:

```
# TYPE ao2_cp_audit_log_appended_total counter
ao2_cp_audit_log_appended_total <N>

# TYPE ao2_cp_audit_log_rotated_total counter
ao2_cp_audit_log_rotated_total <N>

# TYPE ao2_cp_audit_log_persistence_errors_total counter
ao2_cp_audit_log_persistence_errors_total <N>

# TYPE ao2_cp_audit_log_dropped_total counter
ao2_cp_audit_log_dropped_total <N>
```

Counter semantics:

- **`ao2_cp_audit_log_appended_total`** â€” every entry passed to
  `AuditLog::append` since process boot. Mirrors the
  `audit_log.total_appended_since_boot` field on
  `/api/v1/status`. Continues to grow after the ring buffer has
  rotated, so it tracks total request volume even when the
  resident buffer (`total_buffered`) is bounded.
- **`ao2_cp_audit_log_rotated_total`** â€” every successful NDJSON
  file rotation. Mirrors `audit_log.persistence.rotation.count`.
  Stays at `0` when rotation is disabled
  (`AO2_CP_AUDIT_LOG_MAX_BYTES` unset).
- **`ao2_cp_audit_log_persistence_errors_total`** â€” every
  persistence write/serialize/rotation failure. Pairs with the
  non-clearing `persistence_last_error()` peek (also exposed
  on `/api/v1/status` as `audit_log.persistence.last_error`):
  the peek shows the most recent message; this counter says
  how often it has happened. Alert on `rate(...) > 0` to
  catch silent breakage between operator polls.
- **`ao2_cp_audit_log_dropped_total`** â€” every ring-buffer
  eviction (append that arrived when the buffer was already
  at `AO2_CP_AUDIT_LOG_CAPACITY`). Non-zero in steady state
  indicates the capacity is undersized for sustained traffic;
  raise it or enable NDJSON persistence so evicted entries
  survive on disk.

### Suggested Prometheus alerts

Two alert expressions cover the two operator-facing failure
modes:

```promql
# Silent persistence breakage: file rotation/write failure.
# Pairs with the persistence_last_error peek on /api/v1/status.
- alert: AuditLogPersistenceErrors
  expr: rate(ao2_cp_audit_log_persistence_errors_total[5m]) > 0
  for: 1m

# Undersized ring buffer dropping live traffic.
# Raise AO2_CP_AUDIT_LOG_CAPACITY or enable NDJSON persistence.
- alert: AuditLogRingDrops
  expr: rate(ao2_cp_audit_log_dropped_total[5m]) > 0
  for: 5m
```

Both alerts are additive to the existing `/api/v1/status`
operator path â€” operators who poll `/api/v1/status` already
see `audit_log.persistence.last_error` and
`audit_log.persistence.rotation.count`. The metrics counters
turn those into machine-graspable signals that Prometheus
can fan out to PagerDuty/Slack/Opsgenie without a custom
poller.

### Implementation details

`AuditLog` gains two new lock-free `AtomicU64` counters
(`total_dropped` and `total_persistence_errors`) initialised
in all three constructors. They are incremented:

- **`total_dropped`**: at the single eviction site in
  `append()`, only when `entries.len() == capacity` *and*
  there is a real prior entry being evicted to make room.
  Capacity-zero appends (no historical entry to lose) do
  not bump the counter â€” operators who deliberately set
  `AO2_CP_AUDIT_LOG_CAPACITY=0` to disable in-memory
  buffering still see `dropped_total = 0`.
- **`total_persistence_errors`**: at all four error sites in
  `persist()`: JSON serialize failure, missing writer when
  persistence is enabled, `write_all`/`flush` failure, and
  rotate failure. Each one bumps the counter exactly once
  per failed append.

The `metrics_handler` reads the four counters from
`state.audit_log` at scrape time into a new
`AuditLogSamples` struct and passes them to `Metrics::render`,
the same scrape-time-sampling pattern already used for
`storage_index_entries`.

## Upgrade guide

- Binaries are drop-in replacements; on-disk data layout is
  unchanged from v0.1.9 (no migration required).
- The Prometheus exposition adds four new counter families;
  existing scrape config does not need to change. The
  `/api/v1/metrics` endpoint is still token-gated by the
  same bearer.
- The four new counters all start at `0` on each process
  restart (Prometheus `counter` semantics). Existing
  recording rules that consume rate(...) continue to work
  across a restart because the counter resets to `0` and
  rate(...) ignores resets.

## Security posture

- OAuth-only authentication; no API-key provider auth.
- The four new counter values reveal traffic volume and
  failure incidence, not request contents â€” bearer tokens,
  request bodies, and individual entries are never embedded
  in the Prometheus exposition.
- `/api/v1/metrics` remains bearer-token gated by the same
  `AO2_CP_API_TOKEN` middleware as the rest of `/api/v1/*`.
- The persistence-error counter pairs with the existing
  non-clearing `persistence_last_error()` peek so operators
  can alert on incident *count* without exposing the error
  *message* over the metrics surface.
