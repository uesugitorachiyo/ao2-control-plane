## What's new

### Operator dashboard for the audit-log ring buffer

v0.1.5 shipped opt-in NDJSON persistence, v0.1.6 surfaced its
health on `/api/v1/status`, and v0.1.7 added size-based
rotation. The remaining gap was an operator-facing surface: the
machine-readable `/api/v1/audit-log` JSON is the right shape for
Hermes/front-ends, but operators needed a token-gated HTML view
they could open from a browser when triaging traffic. v0.1.8
closes that loop with two paired surfaces over the same buffer:

```
GET /api/v1/audit-log/dashboard           # HTML — operator view
GET /api/v1/audit-log/dashboard.json      # JSON — Hermes/front-end
```

Both accept the same query filters as `/api/v1/audit-log`
(`limit`, `since_unix_micros`, `method`, `status`,
`status_class`, `path_prefix`, `authenticated`). The HTML
surface renders four telemetry cards (ring buffer, persistence,
rotation, filtering), a status-class-coloured table of the
filtered recent entries, and an explicit trust-boundary footer
re-stating that the control plane is a read-only observer and
does not approve or mutate AO artifacts.

The JSON twin emits the identical telemetry under a `buffer`
block alongside `links`, `trust_boundary`, and `entries` so
dashboards can poll either surface without observer divergence:

```json
{
  "schema_version": "ao2.cp-audit-log-dashboard.v1",
  "links": {
    "ring_buffer": "/api/v1/audit-log",
    "status": "/api/v1/status"
  },
  "trust_boundary": {
    "role": "read_only_observer",
    "mutates_ao_artifacts": false,
    "release_approval_owner": "factory-v3 evaluator-closer"
  },
  "buffer": {
    "capacity": 1024,
    "buffered": 42,
    "total_appended_since_boot": 13371,
    "persistence": {
      "enabled": true,
      "path": "/var/lib/ao2-cp/audit.ndjson",
      "last_error": null,
      "rotation": { "max_bytes": 10485760, "count": 3, "last_rotated_unix_micros": 1717104923000000 }
    }
  },
  "entries": [ /* filtered audit-log entries */ ]
}
```

### HTML-injection defence

Path strings rendered into the HTML are HTML-escaped at write
time via a per-handler local helper. Critically, axum keeps
percent-encoded paths verbatim — a request to `/%3Cscript%3E/x`
appears in the audit log as the literal string `%3Cscript%3E/x`,
not as `<script>/x`, so the `<` character never enters the
buffer at all. The dashboard's HTML-injection regression test
drives a `<script>` payload through both the percent-encoded
path *and* a raw `?x=<script>` query string and asserts no
`<script>` substring appears anywhere in the rendered HTML body.
Bearer-token values are never copied into either surface.

## Upgrade guide

- Binaries are drop-in replacements; on-disk data layout is
  unchanged from v0.1.7 (no migration required).
- The two new routes are additive — no existing endpoint or
  JSON shape changes between v0.1.7 and v0.1.8.
- Both new routes require `Authorization: Bearer $AO2_CP_API_TOKEN`,
  matching every other `/api/v1/*` surface.

## Security posture

- OAuth-only authentication; no API-key provider auth.
- The bearer token value is never copied into the HTML dashboard
  or the JSON dashboard payload. The
  `audit_log_dashboard_html_renders_buffer_telemetry_and_redacts_token`
  and `audit_log_dashboard_json_exposes_schema_links_and_redacts_token`
  integration tests assert this on each authenticated render.
- The
  `audit_log_dashboard_html_does_not_render_injectable_script_tag`
  integration test drives a `<script>` payload through both a
  percent-encoded path and a raw query string, then asserts no
  `<script>` substring appears in the rendered HTML body.
- Bearer-required on both surfaces; the
  `audit_log_dashboard_requires_bearer_token` integration test
  asserts a `401` on both HTML and JSON when no token is supplied.
